# UXライティング

デジタルプロダクトとエクスペリエンスのための、明確で簡潔なユーザー中心のインターフェースコピー（UXテキスト/マイクロコピー）を書きます。このスキルは、ユーザーが目標を達成するのを助けるテキストを作成するためのフレームワーク、パターン、ベストプラクティスを提供します。

**互換性:** Claude Desktop、Claude Code、Codex（CLIおよびIDE拡張機能）

**注意:** このスキルはCodex CLI/IDEで動作しますが、ChatGPTでは動作しません。ChatGPTはスキルをインストールまたは使用できません。

## このスキルを使用するタイミング

以下の場合にこのスキルを使用します：
- インターフェースコピーの執筆（ボタン、ラベル、タイトル、メッセージ、フォーム）
- 既存のUXテキストを明確さと効果のために編集
- エラーメッセージ、通知、成功メッセージの作成
- 会話フローやオンボーディングエクスペリエンスの設計
- プロダクトのボイスとトーンの確立
- 一貫性とユーザビリティのためのプロダクトコンテンツの監査

## UXライティングの基本原則

### 4つの品質基準

すべてのUXテキストは以下を満たすべきです：

1. **目的性（Purposeful）** — ユーザーまたはビジネスの目標達成を助ける
2. **簡潔性（Concise）** — 意味を失わずに最小限の言葉を使う
3. **会話性（Conversational）** — 自然で人間的に聞こえ、ロボット的でない
4. **明確性（Clear）** — 曖昧さがなく、正確で理解しやすい

### 主要なベストプラクティス

**簡潔性**
- 1行あたり最大40〜60文字を使用
- すべての言葉に役割が必要
- 密度の高いテキストをスキャン可能なチャンクに分割
- 重要な情報を前に配置

**明確性**
- 平易な言葉を使用（一般向けは中学1年生レベル、専門家向けは高校1年生レベル）
- 専門用語、慣用句、技術用語を避ける
- 全体を通して一貫した用語を使用
- 意味のある具体的な動詞を選択

**会話的なトーン**
- 話すように書く
- 85%の時間で能動態を使用
- 前置詞と冠詞を含める
- ロボット的な表現を避ける

**ユーザー中心**
- 機能ではなくユーザーのメリットに焦点を当てる
- ユーザーの質問を予測して答える
- 二人称（「あなた」）の言葉を使用
- ユーザーの言葉とメンタルモデルに合わせる

## UXテキストパターン

インターフェース要素に以下の一般的なパターンを適用します。

### タイトル
- **目的**: ユーザーが今どこにいるかを示す
- **形式**: 名詞句、文頭のみ大文字
- **種類**: ブランドタイトル、コンテンツタイトル、カテゴリタイトル、タスクタイトル
- **例**: 「アカウント設定」「あなたのライブラリ」「新規投稿を作成」

### ボタンとリンク
- **目的**: ユーザーがアクションを起こせるようにする
- **形式**: 能動的な命令形動詞、文頭のみ大文字
- **パターン**: `[動詞] [目的語]` 
- **例**: 「変更を保存」「アカウントを削除」「詳細を表示」
- **避けるべき**: 「OK」「送信」「ここをクリック」のような一般的なラベル

### エラーメッセージ
- **目的**: 問題を説明し、解決策を提供する
- **形式**: 共感的、明確、実行可能
- **パターン**: `[失敗したこと]. [理由/文脈]. [すべきこと].`

**エラーメッセージの種類**

**バリデーションエラー（インライン）**
- ユーザーがフィールドを完了したとき、またはフォーカスを外したときに表示
- 入力を修正するための簡潔で具体的なガイダンス
- パターン: `[フィールド] [具体的な要件]`
- 例:
  - 「メールアドレスに@を含めてください」
  - 「パスワードは8文字以上必要です」
  - 「未来の日付を選択してください」
- タイミング: リアルタイムまたはフィールド離脱時
- 場所: フィールドの下または横

**システムエラー（モーダル/バナー）**
- バックエンド操作が失敗したときに表示
- 何が起こったか、なぜ起こったかを説明
- パターン: `[失敗したアクション]. [考えられる原因]. [回復手順].`
- 例:
  - 「支払いに失敗しました。カードが拒否されました。別の支払い方法をお試しください。」
  - 「変更を保存できませんでした。接続が切れました。再接続してもう一度お試しください。」
  - 「アップロードに失敗しました。ファイルが大きすぎます。10MB以下のファイルを選択してください。」
- タイミング: 失敗直後
- 場所: モーダルダイアログまたは目立つバナー

**ブロッキングエラー（全画面）**
- 解決するまで継続使用を防ぐ
- ブロッカーと解決方法の明確な説明
- パターン: `[ブロックされているもの]. [理由]. [必要な具体的アクション].`
- 例:
  - 「アップデートが必要です。このバージョンはサポートされていません。続行するには今すぐアップデートしてください。」
  - 「サブスクリプションが期限切れです。アカウントが一時停止されています。アクセスを復元するにはサブスクリプションを更新してください。」
  - 「確認が必要です。機能にアクセスするにはメールアドレスを確認してください。受信トレイを確認してください。」
- タイミング: アプリ起動時または機能アクセス時
- 場所: 全画面または大きなモーダル

**権限エラー**
- 権限を要求する前にメリットを説明
- パターン: `[ユーザーのメリット]. [必要な権限].`
- 例:
  - 「注文が発送されたときに通知を受け取れます。通知を有効にしてください。」
  - 「近くの店舗を見つけられます。位置情報へのアクセスを許可してください。」
  - 「写真をバックアップできます。ストレージの権限を付与してください。」
- タイミング: 機能を初めて使用するとき
- 場所: 機能のコンテキスト内

**避けるべきこと**
- 説明のない技術コード（「エラー403」）
- 非難する言葉（「無効な入力」「不正な文字」）
- ロボット的なトーン（「エラーが発生しました」）
- 行き止まり（回復パスのないエラー）
- 曖昧な原因（「問題が発生しました」）

### 成功メッセージ
- **目的**: アクションの完了を確認
- **形式**: 過去形、具体的、励ます
- **パターン**: `[アクション] [結果/メリット]`
- **例**: 「変更を保存しました」「メールを送信しました」「プロフィールを更新しました」

### 空状態
- **目的**: コンテンツがないときにユーザーをガイド
- **種類**: 初回使用、ユーザーがクリア、エラー/結果なし
- **形式**: 説明 + 入力を促すCTA
- **例**: 「まだメッセージがありません。会話を始めてチームとつながりましょう。」

### フォームフィールド
- **ラベル**: 入力を説明する明確な名詞句（「メールアドレス」「電話番号」）
- **指示**: 動詞を先頭に、なぜ情報が必要かを説明
- **プレースホルダー**: 控えめに使用、「name@example.com」のような標準的な入力のみ
- **ヘルパーテキスト**: 重要度に基づいて、静的、オンデマンド、または自動

### 通知
- **目的**: タイムリーで価値のある情報を提供
- **種類**: アクション必須（侵入的）、受動的（侵入性が低い）
- **形式**: 動詞を先頭にしたタイトル + 文脈的な説明
- **例**: 「アップデートが必要です。続行するには最新バージョンをインストールしてください。」

## ボイスとトーン

### ボイス（一貫したブランドパーソナリティ）
ボイスはプロダクトの一貫したパーソナリティです。以下を使用してボイスを確立します：
- **コンセプト**: 3〜5つの主要なブランド原則/価値観
- **ボイスの特性**: 各コンセプトを説明する形容詞
- **良い例/悪い例**: ボイスを実際に示す具体的な例

ボイスチャートの作成については references/voice-chart-template.md を参照してください。

### トーン（コンテキストに適応）
トーンは、ボイスが特定の状況にどのように適応するかです。ボイスは一定のままですが、トーンはユーザーのコンテキストと感情状態に基づいて変化します。

**トーンの変数**
- **目的**: ユーザーがこのテキストを見る理由（情報、アクション、確認）
- **コンテキスト**: ユーザーが何をしようとしているか（学習、タスク完了、エラーからの回復）
- **感情状態**: ユーザーがどう感じているか（イライラ、興奮、混乱、慎重）
- **重要度**: アクションの影響（低：テーマ変更、高：アカウント削除）

**ユーザーの感情状態によるトーンの適応**

**イライラしている**（エラー、失敗、ブロッカー）
- 共感的で解決策重視
- 非難せずに問題を認識
- 明確な回復パスを提供
- 例: 「支払いに失敗しました。カードが拒否されました。別の支払い方法をお試しください。」

**混乱している**（初回使用、複雑な機能）
- 忍耐強く説明的
- ステップを明確に分解
- コンテキストとガイダンスを提供
- 例: 「銀行を接続して支出インサイトを確認できます。手順をご案内します。」

**自信がある**（日常的なタスク、再訪問）
- 効率的で直接的
- 最小限の説明
- 素早い確認
- 例: 「保存しました」

**慎重**（重要度の高いアクション、データ損失）
- 真剣で透明性がある
- 明確な結果
- ユーザーの決定を尊重
- 例: 「アカウントを削除しますか？すべてのデータが失われ、元に戻せません。」

**成功**（完了、達成）
- ポジティブで励ます
- 達成に比例
- 簡潔な祝福
- 例: 「プロフィールを更新しました。変更が公開されています。」

**コンテンツタイプによるトーンの適応**

**エラーメッセージ**: 共感的、安心させる、解決策重視
- ユーザーを決して非難しない
- 何が起こったかを説明
- 明確な次のステップを提供

**成功メッセージ**: ポジティブ、具体的、励ます
- 何が起こったかを確認
- アクションの重要性に比例
- 簡潔で明確

**指示**: 明確、直接的、役立つ
- 主要なアクションを前に配置
- 必要に応じて理由を説明
- シンプルなステップを使用

**オンボーディング**: 招待的、励ます、簡潔
- 圧倒せずに歓迎
- 価値に焦点を当てる
- 早期の成功を祝う

**確認**: 真剣、透明性がある、尊重する
- 結果について明確
- 操作なし
- 簡単に戻れる

**空状態**: 希望的、実行可能、ガイドする
- なぜ空なのかを説明
- 明確な次のアクションを提供
- 励ますトーンを維持

## 編集プロセス

UXテキストを4つのフェーズで編集します：

### フェーズ1: 目的性
- テキストはユーザーの目標達成を助けるか？
- テキストはビジネス目標に貢献するか？
- ユーザーへの価値は明確か？
- 懸念は予測され対処されているか？

### フェーズ2: 簡潔性
- 不要な言葉を削除
- 冗長な情報を統合
- すべての言葉がその場所に値することを確認
- 重要な概念を前に配置

### フェーズ3: 会話性
- 声に出して読む—これを実際に言うか？
- 能動態を使用（受動態の方が明確な場合を除く）
- 自然な接続詞を含める
- 企業的な専門用語を避ける

### フェーズ4: 明確性
- 具体的で正確な動詞を使用
- 一貫した用語を維持
- 可読性をテスト（Hemingway Editor、Flesch-Kincaid）
- 曖昧さのない意味を確保

## ワークフロー

1. **コンテキストを理解**
   - ユーザーの目標とニーズ
   - ビジネス目標
   - 技術的制約
   - ユーザーの感情状態

2. **コンテンツを下書き**
   - 会話から始める（何と言うか？）
   - 適切なパターンを適用
   - ボイスとトーンを考慮
   - 重要な情報を前に配置

3. **反復的に編集**
   - フェーズ1: 目的性
   - フェーズ2: 簡潔性
   - フェーズ3: 会話性
   - フェーズ4: 明確性

4. **テストと測定**
   - チームでレビュー
   - 可能な限りユーザーでテスト
   - タスク完了率、理解度を測定
   - フィードバックに基づいて反復

## UXライティングにおけるアクセシビリティ

アクセシブルなコンテンツを書くことで、支援技術を使用するユーザーを含むすべてのユーザーが、プロダクトを理解し操作できるようになります。

### アクセシビリティの基本原則

**スクリーンリーダーの最適化**
- すべてのインタラクティブ要素に明示的なラベルを付ける（「送信」だけでなく「フォームを送信」）
- 説明的なリンクテキストを書く（「ここをクリック」ではなく「料金詳細を読む」）
- スクリーンリーダーで機能するようにエラーメッセージを構造化（エラー + フィールドラベルを一緒に読む）
- 視覚的コンテキストが不十分な場合はARIAラベルを使用

**認知的アクセシビリティ**
- 1文あたり8〜14語を目標（8語 = 100%理解、14語 = 90%理解）
- 複雑な情報をスキャン可能なチャンクに分割
- 明確な見出しと論理的な階層を使用
- 一貫性があり予測可能なパターンを提供

**マルチモーダルコミュニケーション**
- 色だけで意味を伝えない
- 視覚的インジケーターとテキストをペアにする（「エラー: メールが必要です」と赤いアイコン）
- アイコンと画像のテキスト代替を提供
- 十分な色のコントラストを確保（WCAG AA最小: 4.5:1）

**すべての人のための平易な言葉**
- 一般向けには中学1〜2年生の読解レベルを目標
- 初めて使用する技術用語を定義
- 慣用句、比喩、文化的参照を避ける
- 一般的で日常的な言葉を使用

### アクセシブルなパターンの例

**ボタン**
- ❌ 悪い: 「送信」（スクリーンリーダーにコンテキストがない）
- ✅ 良い: 「申請を送信」

**リンク**
- ❌ 悪い: 「詳細情報はここをクリック」
- ✅ 良い: 「プライバシーポリシーを読む」

**エラーメッセージ**
- ❌ 悪い: 「無効」と表示される赤いテキスト
- ✅ 良い: 「エラー: メールアドレスに@を含めてください」（エラーアイコン付き）

**フォームラベル**
- ❌ 悪い: プレースホルダーのみのフィールド
- ✅ 良い: 表示されるラベル + オプションのプレースホルダー

## UXテキストのベンチマーク

効果的なUXテキストを作成するために、これらの研究に基づく指標を使用します。

### 文の長さの目標

**コンテンツタイプ別**
- **ボタン/CTA**: 2〜4語が理想、最大6語
- **タイトル**: 3〜6語、最大40文字
- **エラーメッセージ**: 12〜18語（解決策を含む）
- **指示**: 最大20語、理想は14語
- **本文コピー**: 1文あたり平均15〜20語
- **通知**: タイトル + 本文で10〜15語

**理解率**
- 8語以下: ユーザー理解度100%
- 14語以下: ユーザー理解度90%
- 25語: 理解度が大幅に低下する前の最大値

### 文字数と行の長さ

**最適範囲**
- **行の長さ**: 最大の可読性のために40〜60文字
- **ボタンラベル**: 15〜25文字
- **ページタイトル**: 30〜50文字
- **通知タイトル**: 35〜45文字

### 読解レベルガイドライン

**対象者別**
- **一般向け**: 中学1〜2年生（Flesch-Kincaid）
- **専門ツール**: 中学3年〜高校1年生
- **技術製品**: 高校1〜2年生
- **専門分野**: 高校2〜3年生（必要な場合のみ）

**テストツール**
- Hemingway Editor: 複雑な文を強調表示
- Readable.com: 複数の可読性スコアを提供
- Microsoft Word: Flesch-Kincaidスコアリング内蔵

## 避けるべき一般的な間違い

- 受動態の過度な使用
- 一般的なボタンラベル（「送信」「OK」）
- エラーメッセージでユーザーを非難
- 真剣なコンテキストでの過度に賢いユーモア
- 一貫性のない用語
- 隠れた指示や説明
- ユーザー言語ではなくシステム指向の言語
- 言葉が多すぎる（十分に簡潔でない）
- ロボット的で企業的なトーン
- 意味を伝えるために色だけに依存
- アクセシブルでないリンクテキストを書く（「ここをクリック」）

## クイックリファレンス

**文頭のみ大文字**: 「変更を保存」（「変更を保存」ではない）  
**ボタンには能動的な命令形**: 「アカウントを削除」（「アカウント削除」ではない）  
**ユーザー重視**: 「ショートカットで時間を節約」（「ショートカットを提供します」ではない）  
**具体的な動詞**: 「削除」（永久削除の場合は「削除」であって「除去」ではない）  
**前置き**: 「パスワードは8文字必要です」（「パスワードには8文字が必要です」ではない）
