# 汎用テスト原則（言語非依存）

実装の書き方（フレームワークの API、ファイル分割の慣習など）は言語・スタックごとに任せる。エージェントが守るべきは次の **3 つの原則** と、テストケース文言の質である。

## 1. テストケースは「仕様」として書く

- **テストの本文**が実装詳細に触れることはある程度避けきれない。それより危険なのは、**テストケース名（describe / it / 表形式のケース名など）に実装詳細が載ること**である。名前は顧客やプロダクトオーナーにそのまま説明できる行為・結果で書く。
- 実装が変わるたびに「仕様書」が書き換わるのは、ケース名がクラス名・メソッド名・内部モジュール名に引っ張られているときに起きやすい。

**避ける例（ケース名）**

- 内部クラス・関数名だけを並べた説明
- 「〜を呼ぶ」「〜が true を返す」だけで、ユーザー価値が読み取れないもの

**望ましい例（ケース名）**

- ユーザーまたはビジネス上の **Given / When / Then** が言葉で追えるもの
- 失敗時に「何がユーザーにとっておかしいか」が名前から分かるもの

## 2. 古典派（London ではなく Detroit / Classical）

- テストは **本番に近い実行環境**で動かすほど効果が高い。モックは **必要最小限** にし、全体をスタブだらけにしない。
- 「モックの呼び出し回数を検証しただけ」のテストは、リファクタに弱く、回 regressions を拾いにくい。可能なら実オブジェクト・実 I/O（テスト用 DB・コンテナ・ローカルスタック）で境界まで通す。
- 外部 API・ライブラリなど、外部で公開されているもののみモックを利用する。

※ 外部サービス課金・不安定なネットワークなど、古典派だけでは難しい箇所は、境界に閉じたフェイクや契約テストと組み合わせる判断は許容する。原則は「本番の振る舞いに近づける」こと。

## 3. テストはテスト対象と同一ファイル

- テストコードは **テスト対象と同じファイル** に置く（リポジトリの慣習として採用している場合）。別ファイルに分けると、仕様と実装の対応が読み手から離れる。

言語によっては「同一ファイルにテストを書く」ための公式・慣用パターン（インライン実行、条件付き import など）がある。フレームワークの推奨に従う。

## Vitest の例（ケース名とスタイル）

以下は **言語・ランタイム固有の書き方の推奨** ではなく、**「ケース名」と「古典派 vs モック過多」の対比** のための例である。

### ケース名：悪い例 → 良い例

```ts
// 悪い例：実装詳細・内部識別子がケース名に出ている
describe("UserBalanceService", () => {
  it("fetchFromRepository が一度だけ呼ばれる", async () => {
    // ...
  });
  it("mapDtoToViewModel が undefined を渡さない", () => {
    // ...
  });
});
```

```ts
// 良い例：顧客・利用シーンで説明できる言葉
describe("ウォレットの残高表示", () => {
  it("入金後、画面に反映された残高が請求可能額と一致する", async () => {
    // ...
  });
  it("通信エラー時は、前回取得分を表示しつつ再試行できる旨を示す", async () => {
    // ...
  });
});
```

### 古典派 vs モック過多（イメージ）

```ts
// 悪い例：協力オブジェクトを全部モックし、実装の配線だけを検証
it("決済処理", async () => {
  const gateway = { charge: vi.fn().mockResolvedValue({ ok: true }) };
  const repo = { save: vi.fn() };
  await runCheckout(gateway, repo, cart);
  expect(gateway.charge).toHaveBeenCalledOnce();
  expect(repo.save).toHaveBeenCalled();
});
```

```ts
// 良い例：テスト用の決済モードやフェイクゲートウェイで、実際のフローを通す
it("カートが空でないとき、支払い完了後に注文が確定状態になる", async () => {
  const app = createTestApp({ payment: "fake-always-succeeds" });
  await app.addToCart(item);
  const order = await app.checkout();
  expect(order.status).toBe("confirmed");
});
```

実際の「テスト用アプリの立ち上げ方」はプロジェクトのインフラに依存する。原則は **本番と同じコードパスをできるだけ通す** こと。

## エージェント向けチェックリスト

- [ ] `describe` / `it`（または同等の表現）が、**顧客向け説明**に耐えるか
- [ ] ケース名が **クラス名・関数名・レイヤー名** に依存しすぎていないか
- [ ] モックは **境界** に閉じ、全体をスタブの検証にしていないか
- [ ] テストが **対象と同一ファイル** に置ける構成か（プロジェクトのルールに従う）
