# 概要

確定した要件定義・コードベースから、具体的な実装計画を TODO タスクに分解する。

DRY・YAGNI・TDD を意識する

## 前提条件

- 1 つのタスクは、原則として「1 回の PR」または「1 つの機能のコミット」で完結するサイズにすること。
- UI の見た目と、ビジネスロジック（状態管理や API 通信）は必ず別々のタスクに分離すること。

## DRY（Don't Repeat Yourself）

これは重複コードを許すな。ということではない。
「知識」あるいは「目的」が同じものについては、共通化するべきということである。

「たまたま似てしまったコード」を安易に共通化すべきではない。慎重になるべし。

## YAGNI（You Aren't Gonna Need It）

将来必要になるかもしれないと予測して実装を過剰にすべきではない。**現状必要なものだけを実装する** ことを心がける。
早すぎる抽象化もコードの健康状態を害する。シンプルに小さなタスクをつくること。

## TDD（テスト駆動開発）

t-wada 氏が提唱する TDD に則ってタスクを分解すること

## 実行ステップ

### Step 1: コンポーネントの分割 (UI Tasks)

要件定義書の UI 仕様をもとに、必要な React コンポーネントを洗い出します。

- 親コンポーネントと子コンポーネントのツリー構造を整理する。
- 各コンポーネントが受け取るべき `Props` の型を定義する。
- _※ここでは見た目と静的なモックデータでの実装のみをタスク化します。_

### Step 2: ロジックの抽出 (Logic Tasks)

データ取得や状態管理のロジックを抽出し、カスタムフックやユーティリティ関数としてタスク化します。

- 必要なカスタムフックの入出力（引数と戻り値）を定義する。

### Step 3: テストケースの洗い出し (Test Tasks)

TDD での実装を前提とし、各 UI・ロジックに対するテストタスクを定義します。

- 正常系だけでなく、要件定義書にあるエッジケース（エラー時など）のテストを含めること。

### Step 4: 依存関係と実装順序の整理

どのタスクから着手すべきか、依存関係（A が終わらないと B が作れない）を整理し、順序立てます。

## 出力フォーマット

cursor の plan モードで出力されるフォーマットに倣ってタスクリストを出力してください。
